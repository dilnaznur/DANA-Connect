## 1) Project overview

### Purpose & main functionality

DANA Connect is a web application that connects **women in STEM & research sciences in Kazakhstan** with:

- **Mentors** (experienced professionals) who can be discovered via a public mentors directory.
- **Research opportunities** posted by mentors that mentees can browse and apply to.
- **Open collaboration projects** that authenticated users can post, browse, and request to join.

Core features observed in the code:

- **Authentication** via Supabase Auth (email/password).
- **Role-based experiences**:
  - Mentors can post and manage research opportunities; review applications; accept/reject candidates.
  - Mentees can browse opportunities; apply with a motivation statement and optional CV upload; track application status.
- **Applications workflow** for research opportunities:
  - Mentee submits an application.
  - Mentor views applications; opening the “Applications” tab marks pending applications as “viewed”.
  - Mentor accepts or rejects; accepting increments filled spots and can close the opportunity.
  - On acceptance, the app triggers an email notification via a Next.js API route using Resend.
- **Projects workflow**:
  - Authenticated users can post a project with tags, contact info, deadline, and member limits.
  - Users can request to join a project; creators can accept/reject requests and close projects when full.
- **Profile management**:
  - Users have a `profiles` row linked to their Supabase Auth user.
  - Optional profile photo upload to a Supabase Storage bucket.
  - Profile editing page.
- **Internationalization (i18n)**:
  - Language switcher with English/Russian/Kazakh stored in `localStorage`.

### Users & roles

Roles are stored in the `profiles.role` field.

- **Mentor**
  - Posts research opportunities.
  - Reviews applications from mentees.
  - Accepts/rejects applications; acceptance can send an email to the mentee.

- **Mentee**
  - Browses open research opportunities.
  - Applies to opportunities with motivation text and optional CV upload.
  - Tracks application status (pending/viewed/accepted/rejected).
  - Can browse and request to join open projects.

There is **no explicit “admin” role** implemented in the current schema and middleware.

## 3) Database architecture (Supabase)

### High-level

DANA Connect uses Supabase Postgres with:

- Custom application tables in the `public` schema (defined in the SQL files in the repo).
- Supabase-managed tables in `auth` and `storage` schemas.

Supabase-managed tables referenced by this app (not fully enumerated here because their schemas are managed by Supabase and can differ across versions/config):

- `auth.users` — canonical user identity table used by Supabase Auth; `profiles.id` references `auth.users.id`.
- `storage.buckets` — bucket registry for Supabase Storage (e.g., `avatars`, `cvs`).
- `storage.objects` — stored files metadata; RLS policies are applied here for avatar/CV uploads.

Row Level Security (RLS) is enabled on the app tables and policies are defined in SQL.

### Important caveat: schema vs code mismatches

While the repository contains SQL schema/migrations, several parts of the application code imply additional/renamed database fields or relations not present in the provided SQL files.

Observed mismatches:

1. `applications.cv_url` column

- The app inserts and reads `applications.cv_url` (optional CV file path) and signs URLs from a `cvs` storage bucket.
- The provided `supabase-schema.sql` **does not include** a `cv_url` column on `applications`.

2. `open_projects` relation

- The app reads/inserts/updates/deletes from a relation named `open_projects`.
- The provided migration creates a table named `projects`, not `open_projects`.

3. `project_requests.requester_id` vs `project_requests.user_id`

- The provided migration defines `project_requests.user_id`.
- The app code uses `project_requests.requester_id`.

Because documentation should reflect the actual running system, treat the SQL in this repo as the **intended baseline**, and verify the live Supabase DB matches the app expectations.

### Database functions, triggers, and RLS

#### Function: `update_updated_at()`

- Trigger function that sets `NEW.updated_at = now()` on UPDATE.

Triggers defined (from SQL files):

- `profiles_updated_at` on `profiles`
- `opps_updated_at` on `research_opportunities`
- `apps_updated_at` on `applications`
- `projects_updated_at` on `projects`
- `project_requests_updated_at` on `project_requests`

#### RLS (Row Level Security)

RLS is enabled on:

- `profiles`
- `research_opportunities`
- `applications`
- `projects`
- `project_requests`

Policies (high level):

- `profiles`
  - Users can manage their own profile (`auth.uid() = id`).
  - Public can read mentor profiles (`role = 'mentor'`).

- `research_opportunities`
  - Public can read open opportunities (`is_open = true`).
  - Mentors can read/insert/update/delete their own opportunities.

- `applications`
  - Mentees can manage their own applications (`mentee_id = auth.uid()`).
  - Mentors can read/update applications for opportunities they own.

- `projects` + `project_requests`
  - Public can read open projects.
  - Creators can manage their projects.
  - Users can manage their own join requests.
  - Creators can read/update requests for their own projects.

#### RPC: `delete_user_account()`

A helper RPC function that:

1. deletes the user’s profile (cascades to related rows),
2. deletes the auth user row.

The UI calls this RPC during “Delete account”, after attempting to remove stored CV files.

### Supabase Storage (buckets)

The app uses two buckets:

- `avatars` (public)
  - Created in `supabase-migration-photo.sql`.
  - Used for profile photo uploads.

- `cvs`
  - Used for optional CV uploads per application.
  - The repo schema includes only a delete policy for `cvs` under `storage.objects` and does not create the bucket or define all required policies.

## 4) System architecture

### Overall pattern

This is a **client-server web application** implemented as a single Next.js project:

- **Client-side UI**: React components (many pages and dashboards are client components for rich interactivity).
- **Server-side rendering/data fetching**: Next.js server components fetch initial data via the Supabase server client.
- **Backend services**: Supabase provides Auth, Postgres, and Storage.
- **Server-side endpoints**: Next.js Route Handlers under `app/api/*` provide application-specific server logic (currently email notification).

This is not a classic MVC monolith; it’s closer to:

- **Next.js App Router** with a blend of server components + client components,
- **BaaS-centric architecture** where most CRUD is performed directly from the client against Supabase with RLS enforcing access.

### Frontend ↔ backend communication

- Client components use `@supabase/ssr` browser client to call:
  - `supabase.from('<table>').select/insert/update/delete(...)` for Postgres.
  - `supabase.storage.from('<bucket>').upload/list/remove/createSignedUrl(...)` for Storage.
  - `supabase.auth.*` for Auth.

- Server components and route handlers use the Supabase server client configured with Next.js cookies to perform server-side queries.

- A dedicated Next.js API route exists for outbound email:
  - `/api/notify-accepted` sends acceptance emails via Resend.

### Authentication & authorization

Authentication:

- Supabase email/password auth.
- Session is stored as cookies; server code and middleware can read the session.

Authorization:

- Role is stored in `profiles.role` (`mentor` or `mentee`).
- `middleware.ts` enforces:
  - All `/dashboard/*` routes require an authenticated user.
  - If user exists but has no profile, redirect to `/auth/complete-profile`.
  - Mentor routes redirect mentees to their correct dashboard and vice versa.

Data access control:

- Enforced primarily by **Supabase RLS policies** in SQL.

### Data storage

- Primary data store: Supabase Postgres.
- File storage: Supabase Storage
  - Avatars stored under `avatars/<userId>/avatar.<ext>`.
  - CVs stored under `cvs/<userId>/<opportunityId>.<ext>`.
